# fukuaRPA 阶段路线

最后更新：2026-07-13

## 发布边界

- 只维护 Windows 10 1809+ x64 与 Windows 11 x64。
- 同一套完整版源码同时维护多文件版与单文件版；精简版不恢复。
- 未明确要求单文件版时只构建完整多文件版；onefile 保持可选同源产物，不进入默认打包。
- 不在启动时联网；未来更新检查也必须由用户主动触发。
- Python UI 与脚本语义保持兼容，原生模块必须始终有可验证的回退路径。

v1.0.12 继续降低新用户第一次打开软件时的认知负担：多目标、凭据、避让、全局/单步循环边界、
低功耗 UI、变量表达式和窗口控件动作统一收进高级模式。简易模式只过滤新选择入口，不删除功能；
已有高级步骤和高级步进方向会原样保留，切换视图不会静默改变脚本语义。“点位序列”更名为
更直观的“自定义点位”。

v1.0.11 完成稳定工作流引用和面向新手的第二轮地基：步骤跳转由稳定 ID 维护，主列表恢复明确的
拖动落点预览；录制事件按双击、组合键、滚轮与拖拽时长归并。新增有界窗口动作与 UI Automation
控件读写，秘密内容使用当前 Windows 用户 DPAPI 加密且不进入方案、导出包、剪贴板或日志。
全量导出不再容忍缺失图片，只读目录自动回退本地用户数据目录；设置提供简易/高级视图，脚本检查
可定位具体步骤。发布门增加 10 轮相对性能基线，后续明显启动或识别退化将阻止打包。

v1.0.4 维护批次统一了设置行的间距、基线、换行高度和滚动范围，修复紧凑按钮、映射状态、
连续删除悬停与下拉框滚轮误触，并把条件断点下沉到带常用预设的“高级调试”。发布边界按
ADR 0016 调整为同源完整版 onedir + onefile；多文件版仍是推荐产物，精简版继续停更。

v1.0.5 对方向一、二完成无大型依赖的组合优化：原生核心不再为每个缩放档位重复创建线程，
而是整次调用使用一个最多 8 线程的有界任务池；快速一个会优先尝试上次成功尺度，并在同一截图
上无条件保留完整范围回退。全匹配语义不变，通用尺度不变特征与机器学习路线暂不引入。

v1.0.6 将单个“上次尺度”扩展为会话内有界多倍率学习，并允许用户填写手动优先倍率。自动策略
根据缩放网格、历史成熟度、倍率分布和覆盖率动态决定历史容量与优先倍率数量；只有自定义模式
才开放固定上限。API 10900 一次接收完整优先列表，失败后继续原任务池的全部剩余尺度。启动链
同时拆成轻量首帧、工作区导入和运行后端初始化三段，并由源码与冻结 EXE 烟雾记录时间点。

## 第四阶段细化

1. 在截图、区域裁剪、模板预处理、原生匹配、OpenCV 回退、点击和日志刷新处加入低开销计时。
2. 建立固定图片集与 1080p/2K/4K、多区域、缩放、灰度场景的可重复基准。
3. 先消除重复截图、重复模板变换和无效 UI 刷新，再评估 DXGI/Desktop Duplication。
4. DXGI 作为可选原生后端接入；初始化或捕获失败立即回退 MSS，不能改变识别结果语义。
5. 发布前比较 CPU、P50/P95 延迟、内存峰值和长时间稳定性，不以单次最快值作为结论。

阶段四在 v1.0.1 完成。当前 DLL 使用 GDI 捕获并在失败时回退 Python MSS；DXGI/Desktop
Duplication 经评估仍属于高兼容风险项，保留为后续可选后端，不阻塞当前稳定发布。

v1.0.3 二次细化把 OpenCV 全局线程数集中限制为最多 2 个，并在离线诊断与视觉基准中报告、
断言实际线程数。13900HX 上对 1/2/4/8 线程的同场景测量没有显示高线程数收益，因此不再让
OpenCV 按逻辑核心数扩张；用户设置的“识别频率”仍控制业务层等待，两者职责分离。

v1.0.8 已完成空闲后首个目标的三阶段快速唤醒：DXGI 通知经分块指纹确认后缩短自适应额外等待，
常用倍率使用新截图专搜，并带入有界轮换的剩余倍率；任何专搜未命中仍立即执行完整搜索。详细
边界与验收场景见 `docs/RECOGNITION_WAKE_PLAN.md`。

v1.0.9 将日志输出集中为可测试的分类策略：三个预设保持稳定，自定义模式可组合运行、步骤、
分支、动作、识别、坐标、参数、耗时、底层后端、画面唤醒/缩放记忆、界面事件和时间；严重错误
与急停始终保留。配置版本 2 负责旧方案迁移，自定义选择参与普通与全量方案往返。

## 第五阶段细化

1. 用户显式点选窗口后，检查器显示进程、标题、类名、层级、权限、UIA 控件类型与可用动作。
2. 单次左键后台操作优先使用 Invoke/Selection/Toggle/DefaultAction，普通失败才回退 PostMessage。
3. UIA COM 对象固定在专用线程；节点、深度、树遍历、队列和调用等待均有明确上限。
4. 超时请求取消尚未开始的动作；结果未知时不发送第二次消息，避免重复点击。
5. 游戏、浏览器画布、DirectX、自绘和高权限窗口明确标记为可能不兼容，不伪装成功、不静默提权。
6. 真实 Win32 Button 烟雾测试必须证明 UIA Invoke 成功且前台窗口不发生变化。

阶段五在 v1.0.2 完成。后台操作失败仍绝不回退真实鼠标；右键和双击继续使用原有
PostMessage 语义，避免把不同动作强行映射成控件默认操作。

v1.0.3 二次细化要求 UIA 优先命中用户明确绑定的子 HWND；没有精确 HWND 时，再按更深层、
更小面积和动作能力排序，并跳过离屏控件。检查器会报告扫描数、候选数和选中理由；源码与
打包后烟雾均必须证明实际触发的是绑定按钮，而不是同一区域内的父窗口或相邻控件。

## 第六阶段细化

1. 断点发生在步骤真正执行前；步骤列表显示“断”标记，F8 可切换当前步骤断点。
2. 支持完成当前步骤后暂停、继续到下一断点和单步越过；停止请求必须立即唤醒暂停线程。
3. 调试暂停时间不计入全局超时，单独执行选中步骤时清除断点，避免无人继续时挂起。
4. 新增“设置变量”和“判断表达式”，提供 loop、step、attempt、elapsed、last_success 等只读变量。
5. 表达式通过 AST 白名单解释，禁止任意代码、文件和网络访问，并限制长度、节点、数值和变量总量。
6. 流程检查器执行确定赋值分析；暂停变量只显示在当前 UI，不进入轨迹、诊断或日志。

阶段六在 v1.0.2 完成。表达式刻意不是完整 Python，新增语法前必须先证明其可静态验证、
资源有界且不会扩大脚本权限。

v1.0.3 二次细化新增条件断点。条件使用同一 AST 白名单表达式，先参与确定赋值分析，再在步骤
执行前求值；条件为假时继续执行，条件为真时以专用原因暂停。变量缺失或表达式运行错误时宁可
暂停并在当前调试界面显示详情，也不静默漏掉断点；详情仍不得进入轨迹、诊断或持久日志。

## 第七阶段细化

1. `BUILD_INFO.json` 记录源码指纹、依赖版本、目标平台、原生 API/能力和构建边界。
2. 发布目录包含 CycloneDX SBOM、第三方声明与许可证汇总，便于离线审计依赖来源。
3. `RELEASE_INFO.json` 记录 EXE、SBOM、构建记录哈希和系统实际返回的 Authenticode 状态。
4. 包内清单供设置页后台校验；额外日志/配置只警告，额外可执行代码视为失败。
5. 包外逐文件清单支持独立 `--check`，缺失、修改或新增文件都必须被发现。
6. 没有作者证书时明确写明未签名，不使用测试证书；启动和自动更新检查均不联网。

阶段七的软件侧工作在 v1.0.2 完成。包内哈希只能发现损坏，不能证明作者身份；正式代码签名
仍需外部受信任证书，证书可用后再进入第九阶段，不阻塞透明的未签名发布。

v1.0.3 二次细化增加最终用户发布说明、EXE/原生 DLL/UIA 辅助 DLL 的独立哈希与真实签名状态、
流式生成的 `fukuaRPA_v1.0.3_portable.zip` 和外部 `SHA256SUMS`。ZIP 逐成员核对外部目录清单，拒绝
路径穿越、大小写冲突、重复/加密/清单外成员与解压大小异常；277 MB、1219 文件的真实发行目录
已完成构建和复核演练。发布门还会扫描全部 EXE/DLL/PYD 的 PE 架构与导入表，要求非系统依赖
全部随包提供；现有完整包的 153 个 PE、1044 条导入引用已通过闭包审计。哈希继续只表示完整性，
不冒充作者身份认证。

## 第八阶段细化

1. DLL API 11200 公开稳定能力位，覆盖 GDI 捕获、多区域、多缩放、灰度/彩色、全匹配、LRU、计时、预算、ABI 元数据、有界任务池、多倍率回退、指定倍率专搜和画面变化监听。
2. 构建记录、离线自检和诊断导出同时报告 API、能力掩码、最近返回码与错误。
3. 工作预算使用固定返回码 `-2`，在截图前拒绝，并按识别签名缓存本次运行的确定性回退。
4. 普通原生失败连续三次后仅在本次运行停用 DLL；成功或权威空结果会清零连续失败计数。
5. DLL 加载、能力或运行失败时自动使用 OpenCV；用户也可在设置中完全关闭 DLL。
6. 发布门重建 `/MT` C++17 DLL，并运行真实屏幕烟雾、视觉基准、耐久和回退测试。

阶段八在 v1.0.2 完成基础路径，并在 v1.0.3 完成 ABI 与发行态验证。API 10700 公开指针位数、
跨语言结构体大小和 x64、`/MT`、C++17、Windows 10 目标编译标志；Python 加载时逐项核对。
构建后再检查 PE 导入表，禁止 VCRUNTIME/MSVCP/UCRT 动态依赖。真实屏幕烟雾同时覆盖灰度、
彩色、多尺度、全部匹配、多区域排除和 OpenCV 坐标对照，并由打包后的 EXE 再运行一次。

v1.0.5 的 API 10800 保留旧入口，新增扩展入口接收并行模式与优先尺度。自动模式按整次调用的
总工作量决定 1-8 个线程，不设置 CPU 核心亲和性；优先尺度只用于快速一个，命中则提前结束，
失败则复用已捕获画面完成其余尺度。详细设计和基线见 `docs/PERFORMANCE_MULTISCALE_PLAN.md`。

v1.0.6 的 API 10900 新增 `wrpa_find_template_ex2`，可一次传入最多 16 个去重后的优先倍率；
API 10800 的单倍率扩展入口和最早的基础入口继续保留。Python 在原生与 OpenCV 路径应用同一
会话记忆顺序，因此 DLL 缺失或被关闭时仍保持功能一致；只有 API 10900 可在一次调用、一次截图
内完成整个优先列表和完整回退。

v1.0.8 保留所有旧入口，并以 API 11000 的 `wrpa_find_template_ex3` 增加“只搜给定倍率”语义；
API 11100 增加不复制整屏的低分辨率 GDI 指纹，API 11200 增加 DXGI 脏矩形变化通知。DXGI 仅作
门铃，必须经分块指纹确认；驱动或远程桌面不兼容时逐层回退，不改变匹配结果。

步骤调度仍由 Python 负责，这是为了保持现有配置、跳转、调试和日志
语义稳定；将整个调度器迁入 C++ 的收益不足以覆盖回归风险，因此不列为当前交付项。

## 后续阶段

### 第九阶段：正式签名与可验证下载

- 获得作者代码签名证书后，对 EXE、DLL 和发布清单签名，并在干净虚拟机验证信任链和时间戳。
- 更新仍由用户主动触发；下载后先核对签名、版本与哈希，再允许替换旧目录。
- 没有证书前不实现“看起来像签名”的替代方案。

### 第十阶段：真实软件兼容矩阵

- 建立 Win10 1809/22H2、Win11 当前版的干净虚拟机，以及 Office、记事本、浏览器标准控件测试集。
- 对 UIA、PostMessage、前台点击、权限差异和自绘画布分别记录可复现结果，不用单一成功率掩盖限制。
- 只有真实案例证明现有后端不足时，再增加针对性的控件适配。

### 第十一阶段：界面维护性与可访问性

- 继续从 `ui/main_window.py` 抽离设置页构建器、热键/映射控制器和发布诊断控制器。
- 增加键盘导航、焦点顺序、高对比度、100%/150%/200% 缩放和多显示器截图回归。
- 拆分以依赖边界和测试能力为准，不为减少行数制造互相引用的 UI mixin。

### 第十二阶段：可选现代截图后端

- 仅在固定基准证明 GDI/MSS 是主要瓶颈后，实验 Windows Graphics Capture 或 DXGI Desktop Duplication。
- 新后端必须可独立关闭，初始化、权限、休眠、锁屏、显示器切换或显卡驱动异常时立即回退。
- 比较 CPU、P50/P95、首帧、显存/内存和长时间稳定性；不以单次最快结果决定默认启用。

## 持续质量门

- 每次发布运行完整测试、`Ruff --select F,E9,E722`、核心基准和耐久测试。
- 每个外部线程和 Hook 都必须可重复停止，关闭程序后不得留下后台进程。
- 后台点击失败不得回退真实鼠标点击；用户坐标与目标窗口身份必须分别保存和验证。
- 构建产物必须在干净目录完成离线自检，并生成逐文件 SHA-256 清单。
- 新配置字段必须经过默认值、迁移、保存、验证、执行和全量导出六条路径检查。
