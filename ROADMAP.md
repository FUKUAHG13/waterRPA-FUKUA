# fukuaRPA 可能的后续方向

最后更新：2026-07-13

本文记录值得探索的方向，方便维护者和贡献者理解项目还有哪些可能性。它不是发布时间表，也不限制分支、Fork 或后续维护者选择不同的功能边界。

## 当前基础

- 模块化 Python UI、步骤调度、配置存储和可选 C++ 图片识别核心已经分离。
- 当前主线面向 Windows 10 1809+ x64 与 Windows 11 x64。
- onedir 与 onefile 使用同一套业务源码和经过验证的运行库删减逻辑。
- 程序启动时不联网，方案、图片、凭据和日志保存在本地。
- 发布工具可以生成构建记录、SBOM、第三方许可证、运行库闭包、便携 ZIP 和 SHA-256 清单。

## 发布可信度

- 获得合适的代码签名证书后，为 EXE、DLL 和发布清单签名，并在干净虚拟机验证信任链与时间戳。
- 保持哈希、签名和来源边界清晰：哈希用于检查完整性，签名用于验证发布者身份。
- 为非技术用户提供更直观的离线诊断结果，同时避免诊断文件包含脚本内容、秘密文本或本地图片路径。

## 真实软件兼容矩阵

- 在不同 Windows 版本、系统缩放、多显示器和标准用户权限下建立可重复的虚拟机测试。
- 分别记录真实鼠标、PostMessage、UI Automation、浏览器画布、自绘控件和高权限窗口的能力边界。
- 只有真实场景证明现有后端不足时，再增加针对性适配，避免用单一成功率掩盖差异。

## 界面与可访问性

- 继续拆分过大的设置页和主窗口控制器，让界面布局、业务状态和系统 Hook 更容易独立测试。
- 增加键盘导航、焦点顺序、高对比度、100%/150%/200% 缩放和多显示器截图回归。
- 让高级功能保持可发现，但不让第一次使用软件的人必须理解调试器、变量或底层识别参数。

## 性能与识别后端

- 继续以固定场景的 CPU、P50/P95、内存峰值和耐久结果评估优化，不以单次最快值决定默认行为。
- 在 GDI/MSS 确认成为主要瓶颈时，再实验 Windows Graphics Capture 或其他截图后端，并保留可关闭与回退路径。
- 图片识别的缓存、倍率学习和并行策略应保持有界，避免用更高 CPU 占用换取不稳定的短时收益。

## 可选产品变体

### 轻量版

贡献者可以探索删除高体积或低使用率功能的轻量构建。变体应明确列出缺失功能、依赖差异、包体积、配置兼容范围和测试矩阵，并尽量复用共享业务模块，避免复制整套源码后长期漂移。

### 旧系统离线版

面向封闭办公电脑的功能冻结版本是一个可评估方向，目标可以优先覆盖 Windows 7 SP1、Windows 8 和 Windows 8.1。它只保留坐标点击、拖拽、滚轮、按键、等待、循环、热键、本地方案和可选基础图片匹配，不包含联网、自动更新或依赖现代系统 API 的高级功能。

Windows XP 与当前 Python 3.14、Qt 6 和现代依赖不兼容。如果未来确有足够需求，更稳妥的方式是单独评估 XP SP3 x86 的原生 Win32 极简实现，而不是降低当前主线的安全性和可维护性。旧系统版在进入开发前需要先确定功能清单、x86/x64 构建方式、静态运行库、真实虚拟机和目标单位的软件准入要求。

### 其他打包与功能版本

便携目录、单文件、轻量、教学或专用场景版本都可以由维护者提出。每个版本都应公开说明它与主线的差异，不应让相同名称的包在功能、数据格式或安全行为上产生隐藏分叉。

## 维护原则

- 用户数据、凭据和脚本内容优先于兼容便利，迁移前保留原文件，不支持的配置不得被旧程序覆盖。
- 后台操作结果未知时不得重复发送动作，也不得静默改用真实鼠标。
- 新线程、Hook、缓存、日志和原生任务必须具有明确上限与可重复关闭路径。
- 二进制发布前执行完整自动化测试、静态检查、原生构建、性能门、耐久测试、冻结启动和依赖闭包验证。
- 新的产品变体应在文档中说明维护者、支持范围和停止支持方式，而不是把个人选择写成整个社区永久不得改变的规则。
