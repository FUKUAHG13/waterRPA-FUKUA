# ADR 0019：分阶段 GUI 启动与重依赖延迟加载

- 状态：已接受
- 日期：2026-07-12

## 决策

入口只同步导入 Qt 基础组件、产品常量和轻量启动窗口。启动窗口完成第一次真实绘制后，后台线程
导入完整主窗口模块；主线程收到类对象后再创建和显示 `RPAWindow`。主窗口以延迟后端模式构造，
运行按钮保持禁用，直到后台完成 OpenCV/NumPy 检查、原生核心状态确认和进程优先级设置。

PyAutoGUI 由线程安全代理在第一次鼠标或键盘动作时导入。UI Automation 后端只在第一次窗口绑定、
检查或后台控件动作时创建。后台线程只导入 Python 类，不实例化 QWidget；所有窗口创建、显示和
信号处理仍在 Qt 主线程。

源码和冻结 EXE 都通过 `--startup-smoke-file` 记录 Python 启动到轻量首帧、工作区导入耗时、
工作区可见、运行后端就绪及后端自身耗时。发布门校验阶段顺序、成功状态和宽松上限，避免把
某台构建机的瞬时快值误当成所有电脑的硬承诺。烟雾通过 `--startup-runtime-dir` 使用临时可写
目录，不能把测试配置、备份或日志留在正式发行目录。

## 原因

旧入口在 `QApplication` 和任何窗口出现前导入主窗口、执行引擎、OpenCV、NumPy、PyAutoGUI 与
UI Automation，用户会看到数秒无反馈。多文件包无法消除 Python 模块初始化成本，但可以先提供
可响应首帧，再按功能需要加载大型或系统相关后端。

## 后果

启动过程多了明确的准备状态，后端就绪前不能执行脚本。任何新增入口级依赖都必须通过启动架构
测试证明不会提前导入完整工作区或四类重后端。若工作区或后端初始化失败，启动烟雾必须写出
失败报告，GUI 必须给出错误信息，不允许静默启用半初始化的运行按钮。
