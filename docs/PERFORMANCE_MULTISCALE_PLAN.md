# 多尺度识别性能路线

最后更新：2026-07-12（v1.0.6）

## 目标与边界

- 降低多缩放图片识别的延迟、CPU 压力和线程抖动。
- 不引入机器学习、ONNX、CUDA 或其他明显增加发布体积的运行库。
- DLL 始终可关闭；加载、能力或调用失败时继续回退 OpenCV/Python。
- “全部匹配”不得因优化省略缩放档位；快速路径预测失败后必须完整回退。

## 方向一：多核调度

旧核心会按缩放档位串行处理，并在每个档位内部临时创建最多 8 个线程。直接再把不同缩放图
分配给不同核心会形成嵌套并行，容易过量创建线程，尤其不适合大小核混合处理器。

v1.0.5 采用整次调用共享的有界任务池：先把每个“缩放档位 × 识别区域”拆成计算量接近的
结果行块，再由一个动态队列交给 1-8 个线程。自动模式按总工作量决定线程数，关闭模式固定
单线程，强制模式尽量使用上限。任务池不设置核心亲和性，由 Windows 调度器处理大小核与电源策略。

旧 DLL 的 `wrpa_find_template` 入口继续存在；API 10800 的 `wrpa_find_template_ex` 接收并行模式
和一个优先倍率，API 10900 的 `wrpa_find_template_ex2` 接收完整优先倍率列表。

## 方向二：减少无效尺度搜索

“先找到相似元素再计算尺寸”并不是一次免费的预处理：要在未知尺寸下可靠找到元素，本身仍需
尺度不变特征或另一套搜索。ORB/SIFT 对小图标、纯色按钮、重复纹理和缩放插值并不稳定；机器学习
方案则会明显增加体积。因此当前不把它作为默认核心。

v1.0.6 使用有界、可解释的时间先验：快速一个在进程内记录同一模板每次成功倍率、匹配质量和
先后次序，通过频率与近期权重选出多个常用倍率。用户填写的手动倍率永远排在自动学习结果前；
不属于当前缩放范围与步长网格的值会忽略。任一优先倍率达到阈值即可提前返回；全部未命中时，
原生核心复用同一张截图和积分图搜索所有剩余档位，OpenCV 回退也按相同顺序执行完整搜索。

自动策略的稳妥、均衡、积极三档不是固定数字映射，只改变覆盖率、动态容量和历史长度的倾向。
算法根据缩放档位数、已观察倍率种类和历史成熟度计算当前上限；只有启用“自定义记忆容量”后，
用户填写的学习倍率上限和历史次数上限才生效。学习历史只存在内存，不跨重启、不进入方案或
全量导出。设置页会显示当前优先列表与算法容量，完全日志仅在列表或容量变化时输出，避免刷屏。

全部匹配忽略预测并始终运行全量搜索。该优化对多个尺寸反复出现的连续点击收益最大，对尺寸
持续变化的场景只增加有界优先阶段，不改变最终搜索范围。

## 基线与验收

同机 `v1.0.4` DLL 已保存在 `.scratch/performance/v1.0.4`。在 9 个缩放档位、320×180 区域、
6×6 灰度模板的 20 次初始对照中，旧核心匹配累计约 132 ms，新任务池约 47 ms，匹配阶段下降
约 64%；整次耗时仍会受 GDI 截图波动影响，因此发布判断同时查看匹配计时、P50/P95 和 CPU/墙钟比。

发布前必须覆盖：单线程/自动/强制多核、单尺度/多尺度、单个与多个优先尺度命中、整个优先列表
失败后的完整回退、全部匹配、多区域、预算前置回退、旧入口兼容、DLL 缺失回退、源码和冻结包
真实窗口烟雾。

## 暂缓路线

- ORB/SIFT 尺度估计：先建立离线小图标与普通照片数据集，证明召回率后再考虑实验开关。
- DXGI/Windows Graphics Capture：只有基准持续证明截图成为主要瓶颈时再做可选后端。
- 线程常驻池：当前调用级线程池已消除逐缩放重复创建；跨调用常驻线程会增加卸载、停止和 DLL
  生命周期风险，先用耐久数据证明线程创建仍是瓶颈再实施。
